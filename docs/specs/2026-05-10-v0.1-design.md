# hyperclaude v0.1 — Design

## 1. Thesis

A Claude Code plugin built around a deliberate division of labor between two AI coding agents:

- **Claude** implements — planning, coding, subagents, agent teams.
- **Codex** reviews — pre-implementation research, plan critique. (Code review is v0.2.)

> **Claude is the builder. Codex is the critic.**

This split also exploits cost asymmetry: Claude subscription is the bigger spend, so use it for heavy lifting; use Codex as a cheaper second-opinion lens.

The plugin is opinionated, personal, and open-source. It is not trying to be an "everything machine" like oh-my-claudecode. Scope is intentionally tight.

## 2. Scope

### v0.1 in-scope

- Three lifecycle gates:
  1. **Pre-implementation research** (`/hyperclaude:hyper-research`) — Codex
  2. **Plan review** (`/hyperclaude:hyper-plan-review`) — Codex
  3. **Claude implementation arm** — agents + skills, no slash command
- Codex bridge (Node script) — the only place that calls `codex` CLI; runs Codex in `--sandbox read-only` mode
- File-based artifacts under `.hyperclaude/{research,plans,reviews}/`
- A small `tests/` directory containing `node:test` units for the bridge — no external test framework

### v0.1 out-of-scope (with rationale)

| Out | Why |
|---|---|
| Nudge / `UserPromptSubmit` hook | Heuristic-driven layer (Korean regex, cooldown state) — not worth the complexity until gate usage validates demand. v0.2 candidate. |
| Code review gate | Two gates is enough to validate the model. Promote to v0.2 if v0.1 sticks. |
| Auto-execute hooks (gates fire on detection) | Hook brain logic is heavy. Validate via opt-in slash commands first. |
| Other models (Gemini, GPT-mini) | Splits the Claude × Codex thesis. |
| Standalone npm CLI binary | Plugin runtime is enough. Avoids OMC's CLI sprawl. |
| HUD statusline | Nice-to-have, not core. |
| Skill auto-extraction | Meta-automation; premature. |
| Multi-language docs, Discord, sponsors | One-person project at v0.1. |
| Magic keyword expansion | Cognitive overhead vs. value. |
| Workflow modes (autopilot / ralph / ultrawork) | Mode sprawl is exactly the OMC trap to avoid. |

### Non-goals (forever)

- Become an everything machine. Gate count must serve the Claude × Codex thesis.
- Fork or replace Claude Code workflow. This is a plugin, not a fork.
- Cater to other users' preferences at the expense of opinionated defaults. Opt-out via env var only.

## 3. Architecture Overview

Three layers (no auto-trigger / hook in v0.1; deferred to v0.2):

```
    ┌───────────────────────────────────────────────┐
    │              User in Claude Code              │
    └───────────────────────┬───────────────────────┘
                            │
    ┌───────────────────────┼───────────────────────┐
    │                       │                       │
┌───▼─────────────┐ ┌───────▼──────────────┐ ┌──────▼──────┐
│ /hyperclaude:    │ │ /hyperclaude:        │ │   Claude    │
│   hyper-research │ │   hyper-plan-review  │ │  impl arm   │
│   → Codex        │ │   → Codex            │ │             │
└───────┬──────────┘ └─────────┬────────────┘ │  agents/    │
        │                      │              │  skills/    │
        └──────────┬───────────┘              └─────────────┘
                   │
       ┌───────────▼────────────┐
       │ .hyperclaude/          │
       │   research/*.md        │
       │   plans/*.md           │
       │   reviews/*.md         │
       └────────────────────────┘
```

| Layer | Role |
|---|---|
| Slash commands | Entry points (`/hyperclaude:hyper-research`, `/hyperclaude:hyper-plan-review`). Provided directly by the named skills — no separate `commands/` directory. Plugin-namespaced per Claude Code's invocation contract. |
| Skills | Gate behaviors + impl discipline. Standard Claude Code SKILL.md. |
| Agents | Claude impl arm — `planner`, `implementer`, `verifier`. Standard subagent definitions. |

**No automatic surface in v0.1.** The earlier nudge / `UserPromptSubmit` hook idea is deferred to v0.2 (see §9). v0.1 is purely opt-in: the user invokes a gate, the gate runs, the user reads the output. This keeps v0.1 small enough to ship and validate before adding any heuristic-driven layer.

External dependencies: Claude Code plugin runtime, `codex-cli >= 0.128.0`, Node 18+. **Zero npm deps** for bridge script (Node stdlib only).

## 4. Components & Directory Layout

```
hyperclaude/
├── .claude-plugin/
│   ├── plugin.json           # plugin metadata only — no skills/commands arrays
│   └── marketplace.json      # self-hosted marketplace entry
├── skills/
│   ├── hyper-research/SKILL.md
│   ├── hyper-plan-review/SKILL.md
│   ├── hyper-tdd/SKILL.md
│   └── hyper-debug/SKILL.md
├── agents/
│   ├── planner.md
│   ├── implementer.md
│   └── verifier.md
├── scripts/
│   └── codex-bridge.mjs
├── templates/
│   └── codex/
│       ├── research.md
│       └── review.md
├── tests/
│   └── codex-bridge.test.mjs
├── docs/
│   ├── architecture.md
│   ├── specs/
│   └── plans/
├── README.md
└── LICENSE
```

`plugin.json` carries metadata (name, version, license, repo) only. Skills and agents are loaded via Claude Code's default-directory auto-discovery — listing them again in the manifest risks double-load and is unnecessary.

**No `commands/` directory.** A skill named `hyper-research` is automatically invokable as `/hyperclaude:hyper-research <args>`; the user-typed argument is part of the message Claude receives and the skill body explains how to extract it. Keeping a separate `commands/hyper-research.md` would shadow the skill (Claude Code resolves the skill first when names collide), and shadowing would lose the skill body's argument-extraction logic.

### Component counts (v0.1)

| Component | Count | Notes |
|---|---|---|
| Skills | 4 | 2 user-invoked gate skills (`/hyperclaude:hyper-research`, `/hyperclaude:hyper-plan-review`) + 2 auto-loaded impl-discipline skills (`hyper-tdd`, `hyper-debug`) |
| Agents | 3 | `planner`, `implementer`, `verifier` |
| Scripts | 1 | `codex-bridge.mjs` |
| Templates | 2 | research + review prompt templates |

### Naming conventions

- All hyperclaude-owned skills use the `hyper-*` prefix with kebab-case throughout: `hyper-research`, `hyper-plan-review`, `hyper-tdd`, `hyper-debug`. No glued forms.
- Slash commands match skill IDs 1:1. Claude Code namespaces plugin commands as `/<plugin>:<command>`, so the actual invocation is `/hyperclaude:hyper-research` and `/hyperclaude:hyper-plan-review`. Short aliases (e.g., dropping the `hyper-` prefix once namespacing handles disambiguation) are deferred until friction is observed.
- Agent IDs (`planner`, `implementer`, `verifier`) intentionally drop the `hyper-` prefix — they're not user-invoked surfaces and the bare names read naturally in `Agent(subagent_type=...)`.

### Artifact directory (in consumer projects)

```
.hyperclaude/
├── research/<YYYYMMDD-HHMM>-<slug>.md
├── plans/<YYYYMMDD-HHMM>-<slug>.md
└── reviews/<YYYYMMDD-HHMM>-<slug>.md
```

`.hyperclaude/` commit policy is the user's choice. The README will note: "If you don't want these artifacts in git, add `.hyperclaude/` to `.gitignore`."

## 5. Data Flows

### Flow 1 — Pre-implementation research (`/hyperclaude:hyper-research`)

1. User: `/hyperclaude:hyper-research add OAuth login to the API`
2. The user's message including the slash and argument is delivered to Claude; the `hyper-research` skill activates by name.
3. Claude invokes the `hyper-research` skill.
4. Skill body: calls `node "${CLAUDE_PLUGIN_ROOT}/scripts/codex-bridge.mjs" research --task "..."`.
5. `codex-bridge.mjs`: loads template, spawns `codex exec --sandbox read-only -`, captures output, writes `.hyperclaude/research/<timestamp>-<slug>.md`, prints `{ok, path}` JSON.
6. Claude reads the file via Read.
7. Claude integrates findings into the next planning step.

### Flow 2 — Plan review (`/hyperclaude:hyper-plan-review`)

1. Earlier in the session, Claude has written a plan to `.hyperclaude/plans/<timestamp>-<slug>.md` (file-based handoff, per superpowers convention).
2. User: `/hyperclaude:hyper-plan-review` (no arg) or `/hyperclaude:hyper-plan-review path/to/plan.md`.
3. Skill resolves which plan to review:
   - If arg given, use it.
   - Else find most-recent file in `.hyperclaude/plans/`.
   - Else error: "no plan found, write one or pass a path."
4. Skill calls `node "${CLAUDE_PLUGIN_ROOT}/scripts/codex-bridge.mjs" review --plan-path <path>`.
5. Bridge writes `.hyperclaude/reviews/<timestamp>-<slug>.md` (same slug as the plan it reviews).
6. Claude reads the review and refines the plan (in-place edit or new versioned file).

### Flow 3 — Claude implementation arm (no slash command)

This flow runs without explicit invocation:

1. Claude has an approved plan.
2. Claude invokes `Agent(subagent_type="planner|implementer|verifier", ...)` per task needs.
3. Skills auto-apply via standard skill description matching:
   - `hyper-tdd` → triggered when Claude is writing implementation code (description language authored in implementation phase).
   - `hyper-debug` → triggered when Claude hits an error or test failure.

The exact body of `hyper-tdd` and `hyper-debug` is **out of scope for this design doc** — they will codify the author's preferred discipline during the implementation plan. Both are inspired by, but not ported from, superpowers' `tdd` and `systematic-debugging`.

**Why no slash command:** explicit invocation defeats the goal of automatic dispatch. Claude already knows the Agent tool; the agent definitions and skills shape its behavior.

## 6. Codex Bridge

### Why isolate the bridge

`scripts/codex-bridge.mjs` is the **only** place that invokes `codex`. Skills never call codex directly. This keeps:
- Prompt assembly + frontmatter format in one file.
- Codex CLI flag changes localized to one edit.
- Skill markdown clean of shell incantations.

### CLI interface

```bash
node scripts/codex-bridge.mjs research --task "<text>" [--slug <slug>] [--out <dir>] [--dry-run]
node scripts/codex-bridge.mjs review   --plan-path <path> [--slug <slug>] [--out <dir>] [--dry-run]
```

`--dry-run` validates args, locates the prompt template, generates the slug, and prints the resolved invocation as JSON without spawning `codex`. Used by smoke tests and by skills wanting a preflight check.

**Stdout contract (consumed by skill via Bash tool):**
```json
{ "ok": true, "path": ".hyperclaude/research/...md", "slug": "oauth-login" }
```
On failure:
```json
{ "ok": false, "error": "codex CLI not found", "hint": "see codex:setup" }
```

### Prompt templates

Live in `templates/codex/{research,review}.md`. Bridge loads, substitutes `{{TASK}}` / `{{PLAN}}` placeholders, sends to codex. Markdown so prompt tuning is git-trackable without JS edits.

### Output file format

```markdown
---
mode: research      # research | review
task: |-
  add OAuth login to the API
slug: oauth-login
generated: 2026-05-10T10:15:00Z
codex-version: 0.128.0
template-version: 1
---

# Research: add OAuth login to the API

[codex raw output]
```

**Frontmatter rules:**

- `task` is **always** a YAML block scalar (`|-` style). This makes the format safe for tasks containing colons, quotes, newlines, or other YAML metacharacters without needing escape logic in the bridge.
- `mode` ∈ `{research, review}`.
- `slug` matches the file basename (sans timestamp + extension) for traceability.
- `codex-version` captured from `codex --version` at run time. If detection fails, set to `unknown`.
- `template-version` increments only when the prompt template's structure changes (not for wording tweaks).

Frontmatter enables traceability (slug-link of research → plan → review) and future v0.2 features (caching, replay).

### Slug derivation

- `--slug` if given.
- Else slugify task: lowercase, first 5 words, kebab-case, **strip non-ASCII** (do not transliterate — keeps the v0.1 slugifier trivial).
- Else timestamp-only filename.

### Codex invocation

Bridge spawns `codex exec --sandbox read-only -` (stdin mode), writes the assembled prompt to stdin, then captures stdout.

```js
const child = spawn(
  'codex',
  ['exec', '--sandbox', 'read-only', '-'],
  { stdio: ['pipe', 'pipe', 'pipe'] }
);
child.stdin.end(promptText);
// capture stdout → write to output file
```

**Why `--sandbox read-only`:** Codex's role in hyperclaude is *critic*, not *implementer*. Read-only sandbox forbids workspace writes regardless of the user's local Codex defaults — preserves the product's role boundary even if Codex is configured permissively elsewhere.

**Why stdin and not `--file`:** `codex-cli 0.128.0` does not implement `--file` for `exec`. Verified against `codex exec --help` on that version.

**Why not positional `codex exec "<prompt>"`:** safe for short prompts but fragile when the prompt contains shell metacharacters or exceeds shell argv limits. Stdin avoids both.

**Version pinning:** spec assumes `codex-cli >= 0.128.0`. Bridge runs `codex --version` once at startup; if missing or older, refuses to run with a clear error and a hint to upgrade.

**Timeout:** Bridge accepts `--timeout` (default 300s). Skill invokes Bash with `timeout: 600000` (10 min) to give margin.

### Error matrix

| Case | Behavior |
|---|---|
| `codex` not found | `{ok:false, error, hint}` |
| codex non-zero exit | save partial output if any, surface stderr |
| `.hyperclaude/` missing | bridge mkdir -p |
| timestamp+slug collision | suffix `-2.md`, `-3.md` (rare; minute-resolution + slug) |
| `--plan-path` missing | hard fail before invoking codex |

### Dependencies

**Zero npm packages.** Node stdlib only (`node:fs`, `node:child_process`, `node:path`). No lockfile in v0.1. Frontmatter is hand-rolled (~30 lines).

### Bridge does not do (in v0.1)

- Caching of identical task → fresh run every time.
- Multi-model fan-out (codex + others).
- Async / queued runs.
- Automatic retry on failure.

## 7. Conventions

### Plan handoff

- Plan files live at `.hyperclaude/plans/<YYYYMMDD-HHMM>-<slug>.md`.
- Claude is taught the convention via README only (v0.1). If 9-of-10 inference rate is insufficient in practice, add a `hyper-plan` convention skill in v0.x.
- `/hyperclaude:hyper-plan-review` finds plans in priority order: explicit arg → most-recent in `.hyperclaude/plans/` → error.

### Slug shape

- Lowercase kebab-case.
- ≤5 words.
- ASCII only — **drop non-ASCII characters** (no transliteration in v0.1, matching the bridge's slugifier in §6).
- Same slug across the trio: `oauth-login` research file, plan file, review file.

### Timestamp format

`YYYYMMDD-HHMM` (e.g., `20260510-1015`) — sortable, human-readable, minute resolution.

## 8. Success Criteria for v0.1

### Acceptance smoke checks (must pass before tagging v0.1)

These are evaluated **after scaffolding is complete** — not against the bare repo at design-approval time. Currently the repo is design-only, so all checks are expected to fail until the implementation plan delivers `.claude-plugin/plugin.json`, `scripts/`, etc.

- [ ] `claude plugin validate .` passes against `.claude-plugin/plugin.json`.
- [ ] `node scripts/codex-bridge.mjs research --task "test" --dry-run` exits 0 and prints `{ok: true, dryRun: true, ...}` without spawning `codex`.
- [ ] `node --test tests/codex-bridge.test.mjs` — all unit tests pass (slugify, frontmatter, args parsing, dry-run, collision suffix).
- [ ] End-to-end on a fresh consumer project: plugin install → `/hyperclaude:hyper-research <task>` writes a file under `.hyperclaude/research/` with valid frontmatter.

### Field success (judged after ~2 weeks of personal use)

- [ ] Author uses `/hyperclaude:hyper-research <task>` naturally at the start of new work.
- [ ] `/hyperclaude:hyper-plan-review` auto-locates the right plan ≥80% of the time.
- [ ] Codex bridge failure rate < 10%.
- [ ] On a fresh project: plugin install → first gate working in <5 minutes.

If acceptance + field success both hold, promote to v0.2.

## 9. v0.2 Candidates

In rough priority order:

1. **Nudge / `UserPromptSubmit` hook** — passive suggestion layer that detects build/plan intent in user messages and recommends a gate. Deferred from v0.1 because (a) the language-detection heuristic is non-trivial (Korean `\b` regex doesn't work as in English), and (b) cooldown state management is a meaningful engineering surface that should be informed by real gate-usage data.
2. **`/hyperclaude:hyper-code-review`** — third gate: code review of diff/PR via Codex.
3. **`hyper-plan` convention skill** — if README-only proves insufficient at teaching the plan-file location convention.
4. **Bridge result caching** — same task within N minutes returns cached file.
